# מסמך אפיון – אפליקציית מעקב ריצה אישית

מסמך זה מיועד להעברה ל-Claude Code כהוראת עבודה לבניית האפליקציה מתחילתה ועד הדפלוי.

## 1. רקע ומטרת האפליקציה

אפליקציית web ידידותית (PWA – ניתנת להתקנה למסך הבית בטלפון), למשתמש/ת יחיד/ה, שמטרתה:
1. להציג תוכנית ריצה של 4 חודשים (ניתנת לעריכה).
2. לאפשר תיעוד ריצות בפועל (מרחק, זמן, קצב, תחושה, הערות).
3. להציג היסטוריה והתקדמות.
4. כפתור "משוב AI" שמפעיל ניתוח של ה-Claude API על הנתונים שנאספו, ומחזיר משוב **אנליטי וענייני בלבד — בלי עידוד, בלי דחיפה, בלי שפה מוטיבציונית**. זו דרישה קריטית: ה-AI צריך להתנהג כמו אנליסט נתונים שמספק תובנות עובדתיות, לא כמאמן מעודד.
5. דף הסברים בתוך האפליקציה על שיטת האימון והרציונל של התוכנית.

שפת ממשק: עברית, RTL, כברירת מחדל בלבד — **כל הטקסטים בממשק** (כפתורים, תוויות, הודעות שגיאה/מצב, וגם תשובות ה-AI) חייבים להיות כתובים בעברית, לא רק כיוון התצוגה.

תאריך התחלה בפועל של התוכנית: 17.6.2026. כלל קבוע: **אין אף פעם אימון ביום שישי** — חשוב לשמור על זה גם אם בעתיד מישהו יוסיף סשנים חדשים לתוכנית.

## 2. דגשים קריטיים (must-have, לא להתפשר עליהם)

- משוב ה-AI חייב להיות נייטרלי-אנליטי. system prompt חייב לכלול הוראה מפורשת לא להשתמש בשפה מעודדת/מוטיבציונית/סימני קריאה נלהבים, ולא "לדחוף" את המשתמש.
- התוכנית חייבת להיות ניתנת לעריכה (שינוי תאריך, מרחק, סוג אימון לכל סשן).
- חוסן לפספוסים: סשן שפוספס לא "שובר" את התוכנית — אפשר לסמן כפוספס, לדחות ליום אחר, או להמשיך כרגיל. אין לוגיקה אוטומטית קשוחה שמוחקת או חוסמת התקדמות.
- מפתח ה-Anthropic API חייב להיות בצד שרת בלבד (משתנה סביבה), לעולם לא חשוף בצד הלקוח.

## 3. סטאק טכנולוגי מומלץ

- **Next.js (App Router) + TypeScript** — frontend + backend (API routes) בקוד בסיס אחד, נוח לדפלוי ב-Railway.
- **Tailwind CSS** לעיצוב.
- **Prisma ORM + PostgreSQL** — Railway מציע תוסף Postgres בקליק.
- **next-pwa** (או חלופה דומה) להפיכת האפליקציה ל-PWA הניתנת להתקנה.
- **Anthropic API** (קריאה מצד שרת בלבד) לפיצ'ר המשוב.
- ניהול גרסאות: Git → GitHub → Railway (auto-deploy מה-branch הראשי).

אין צורך באימות משתמשים (login) — זו אפליקציה לאדם אחד. אם יש חשש מפרטיות כי ה-URL ציבורי, אפשר (אופציונלי, לא חובה ב-MVP) להוסיף שכבת הגנה פשוטה של סיסמה אחת בקובץ middleware.

## 4. מודל נתונים (Prisma schema – סקיצה)

```prisma
model PlanSession {
  id            String   @id @default(cuid())
  monthNumber   Int
  weekNumber    Int
  dayLabel      String   // לדוגמה "אימון קצר 1", "אימון ארוך", "יום גמיש"
  plannedDate   DateTime
  targetKm      Float
  methodNote    String?  // הסבר שיטת ריצה-הליכה לאותו סשן
  status        String   @default("planned") // planned | done | skipped | rescheduled
  linkedRunId   String?  // אם הושלם, קישור ל-RunLog
}

model RunLog {
  id            String   @id @default(cuid())
  date          DateTime
  distanceKm    Float
  durationMin   Float
  paceMinPerKm  Float    // מחושב: durationMin / distanceKm
  feeling       Int?     // 1-10
  notes         String?
  planSessionId String?
}
```

## 5. עמודי האפליקציה

1. **בית/דשבורד** — הסשן הקרוב בתוכנית, סטטיסטיקה מהירה (ק"מ השבוע, % עמידה בתוכנית), כפתור "משוב AI" בולט.
2. **התוכנית** — תצוגת כל 4 החודשים לפי שבועות, עם אפשרות עריכה (תאריך/מרחק/סטטוס) לכל סשן.
3. **תיעוד ריצה** — טופס: תאריך, מרחק, זמן (קצב מחושב אוטומטית), תחושה (1-10), הערות, אופציה לקשר לסשן מהתוכנית.
4. **היסטוריה** — רשימת ריצות + גרף התקדמות מרחק/קצב לאורך זמן.
5. **הסברים** — תוכן סטטי: שיטת ריצה-הליכה, הרציונל של בניית התוכנית (5/10/15 ק"מ), מילון מושגים (קצב, RPE וכו').
6. **משוב AI** — מודאל/דף שמציג את תשובת ה-AI כטקסט.

## 6. פיצ'ר המשוב מבוסס-AI – אפיון מדויק

- Endpoint: `POST /api/feedback`
- השרת אוסף: נתוני התוכנית, יומן הריצות האחרון (למשל 10-15 רשומות אחרונות), ואחוז עמידה בתוכנית.
- קריאה ל-Anthropic Messages API מהשרת בלבד, עם `ANTHROPIC_API_KEY` ממשתנה סביבה.
- **System prompt (לשלב בקוד, בעברית), חייב לכלול:**
  - "אתה מנתח נתוני ריצה. תפקידך לתת משוב עובדתי ואנליטי בלבד על ההתקדמות לפי הנתונים שיוצגו לך."
  - "אל תשתמש בשפה מעודדת, מוטיבציונית, או נלהבת. אל תגיד 'כל הכבוד', 'אתה מצוין', 'תמשיך כך' וכל ביטוי דומה."
  - "התייחס לעובדות בלבד: עמידה בתוכנית, מגמות בקצב, מגמות במרחק, פערים בין מתוכנן לבוצע. אפשר להציע התאמות קונקרטיות (למשל הפחתת קצב, יום מנוחה נוסף) על בסיס הנתונים בלבד."
  - "אל תשתמש בסימני קריאה."
- התשובה מוצגת כטקסט רגיל למשתמש.

## 7. דרישות PWA

- `manifest.json` עם שם אפליקציה, אייקונים (192px, 512px), צבע נושא.
- Service worker (דרך next-pwa) לקאשינג נכסים סטטיים, כדי שתהיה אפשרות "הוספה למסך הבית" באנדרואיד/iOS Safari.
- בדיקה ידנית: פתיחה בכרום נייד → תפריט → "הוסף למסך הבית".

## 8. הנחיות UI/UX

- RTL בעברית כברירת מחדל בכל הדפים.
- mobile-first, כפתורים גדולים ונגישים למגע.
- פלטת צבעים רגועה וספורטיבית (לא צעקנית) — למשל גווני כחול/ירוק עם הדגשות.
- ניווט תחתון (bottom nav) בנייד: בית | תוכנית | תיעוד | היסטוריה | הסברים.
- הודעות מצב ברורות (למשל "סשן זה פוספס – ניתן לדחות ליום גמיש") בלי טון שלילי.

## 9. שלבי בנייה (להוראה ל-Claude Code, בסדר הזה)

1. אתחול פרויקט: `Next.js + TypeScript + Tailwind`, `git init`, קובץ `.env.example` עם `DATABASE_URL` ו-`ANTHROPIC_API_KEY`.
2. הגדרת Prisma schema (סעיף 4) + מיגרציה ראשונה.
3. סקריפט seed שמכניס לטבלת `PlanSession` את כל הסשנים של 4 החודשים, **עם התאריכים המדויקים** (לוח שנה אמיתי, לא תאריכים יחסיים) המפורטים בטבלאות של מסמך "תוכנית ריצה 4 חודשים" (קובץ נפרד). יש להזין כל שורה בדיוק כפי שמופיעה שם — תאריך, סוג אימון (קצר1/קצר2/ארוך/יום גמיש), ומרחק מטרה. שים לב: ימי שישי לא מופיעים בתוכנית בכלל.
4. API routes: CRUD לתוכנית (`/api/plan`), CRUD ליומן ריצות (`/api/runs`), ו-`/api/feedback` כמתואר בסעיף 6.
5. בניית העמודים (סעיף 5) עם Tailwind, RTL, ניווט תחתון.
6. הגדרת PWA (manifest + service worker + אייקונים).
7. אינטגרציית AI feedback מצד שרת בלבד, בדיקה שהמפתח לא נחשף ב-bundle של הלקוח.
8. בדיקות מקומיות: יצירת/עריכת סשן, הוספת ריצה, חישוב קצב נכון, גרף היסטוריה, קריאת AI feedback.
9. `git add` + `commit` + push לריפו GitHub חדש (לפני כן `.gitignore` ל-`node_modules`, `.env`).
10. דפלוי ל-Railway: חיבור הריפו, הוספת תוסף PostgreSQL (זה ייצר את `DATABASE_URL` אוטומטית), הגדרת `ANTHROPIC_API_KEY` כמשתנה סביבה ידנית בהגדרות הפרויקט, הגדרת build/start commands (`next build` / `next start`), ואז בדיקת ה-URL החי.
